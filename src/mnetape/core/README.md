# Core

The `core` package contains all non-UI logic: data models, code generation, pipeline execution, project state, and QC reporting.

## Modules

| Module              | Purpose                                                                   |
|---------------------|---------------------------------------------------------------------------|
| `models.py`         | `ActionConfig`, `ActionStatus`, `ActionResult`, `DataType`, `ICASolution` |
| `data_io.py`        | EEG file loading and format detection                                     |
| `data_store.py`     | Disk-backed checkpoint store for per-step intermediate data               |
| `executor.py`       | Executes a single action against live MNE objects                         |
| `codegen.py`        | Bidirectional mapping between `ActionConfig` lists and Python script text |
| `project.py`        | `Project`, `Participant`, `Session` models and JSON serialization         |
| `qc_report.py`      | HTML QC report generation for processed sessions                          |
| `ast_utils.py`      | AST helpers used by codegen and the `@builder` decorator                  |
| `logging_config.py` | Logging setup                                                             |

## Pipeline code flow

1. The GUI holds an ordered `list[ActionConfig]`.
2. `generate_full_script(actions, extra_preamble=None)` produces a runnable Python script.
3. `exec_action(call_site, func_defs, data, action, input_type, output_type)` executes one action at a time, returning the updated data object.
4. `parse_script_to_actions(script)` reconstructs an `ActionConfig` list from an edited script.
5. `pipeline_canonical_code(actions)` produces the script with participant-specific values (e.g. file paths) cleared (used for pipeline hashing and staleness detection).

The script format uses named functions so the generated code is human-readable and directly reusable outside the app:

```python
# EEG Preprocessing Pipeline
# Auto-generated by the MNETAPE software.
import mne
import numpy as np

# --- Functions ---
def filter(raw, l_freq, h_freq):
    raw.filter(l_freq=l_freq, h_freq=h_freq)
    return raw

# --- Pipeline ---
# [1] Bandpass Filter
raw = filter(raw, l_freq=1.0, h_freq=40.0)
```

Custom inline blocks written directly in the code editor are preserved verbatim through parse/generate round-trips.

## Models

### `DataType`

Enum identifying the MNE data object currently flowing through the pipeline:

| Value    | MNE type             | Scope variable(s) |
|----------|----------------------|-------------------|
| `RAW`    | `mne.io.Raw`         | `raw`             |
| `EPOCHS` | `mne.Epochs`         | `epochs`          |
| `EVOKED` | `mne.Evoked`         | `evoked`          |
| `ICA`    | `ICASolution` bundle | `ica`, `raw`      |
| `ANY`    | pass-through         | (unchanged)       |

`DataType.ANY` is used for type-variant actions that have different bodies for each input type. The actual type flowing through is tracked externally by `get_types_for_actions`.

### `ICASolution`

```python
@dataclass
class ICASolution:
    ica: mne.preprocessing.ICA
    raw: mne.io.Raw
```

Bundles a fitted ICA decomposition with the raw data it was fitted on. Flows through the pipeline as `DataType.ICA`.
`ica_fit` produces it; `ica_apply` consumes it and returns `Raw`.

### `ActionStatus`

Per-step execution status, stored in `ActionConfig.status`:

| Value      | Meaning                       |
|------------|-------------------------------|
| `PENDING`  | Not yet executed              |
| `COMPLETE` | Executed successfully         |
| `ERROR`    | Execution failed              |

### `ActionResult`

```python
@dataclass
class ActionResult:
    summary: str
    fig: Figure | None = None
    details: dict = field(default_factory=dict)
```

Optional structured feedback an action can return after execution (summary text, a matplotlib figure, and key/value details).
This result can be displayed by the GUI after execution.

### `ActionConfig`

Mutable runtime configuration for one pipeline step:

| Field             | Type                   | Purpose                                                                |
|-------------------|------------------------|------------------------------------------------------------------------|
| `action_id`       | `str`                  | Key into the action registry                                           |
| `params`          | `dict`                 | Primary parameter values                                               |
| `status`          | `ActionStatus`         | Execution state                                                        |
| `error_msg`       | `str`                  | Last error message, if any                                             |
| `custom_code`     | `str`                  | User-edited body; replaces generated code when non-empty               |
| `is_custom`       | `bool`                 | True when `custom_code` was set by the user                            |
| `title_override`  | `str`                  | Display name for custom actions; set from `# [N] Title` during parsing |
| `advanced_params` | `dict`                 | Extra MNE kwargs grouped by kwargs group name                          |
| `result`          | `ActionResult \| None` | Feedback produced after execution                                      |

## Code generation (`codegen.py`)

### Script generation

`generate_full_script(actions, extra_preamble=None) -> str`

Produces the complete Python script. Key steps:
1. `get_types_for_actions(actions)`: tracks the `DataType` at each step.
2. `assign_func_names(actions, types)`: assigns unique function names; actions with identical bodies share one `def`.
3. `collect_func_defs(actions, func_names, types)`: builds the `# --- Functions ---` section.
4. Call sites are written in `# --- Pipeline ---`.

Extra imports required by specific actions (e.g. `from scipy import signal`) are pulled from `ActionDefinition.extra_imports` and deduplicated.

### Script parsing

`parse_script_to_actions(script) -> list[ActionConfig]`

Reconstructs the action list from an edited script. Parsing strategy per action block:
- Lookup by title first, then by function name (strips dedup suffix like `_2`).
- If the function body in the script differs from all canonical bodies, the body is stored as `custom_code`.
- Multi-line or unresolvable blocks become `custom` actions.

`parse_call_site(call_site, action_def) -> (params, advanced_params, func_name)` parses kwargs back into primary and advanced groups.

### Canonical code

`pipeline_canonical_code(actions) -> str`

Same as `generate_full_script` but clears participant-specific values (e.g. `load_file`'s `file_path`).
Used for MD5 hashing to detect whether a participant's processed output is stale relative to the current pipeline.

## Execution (`executor.py`)

`exec_action(call_site, func_defs, data, action, input_type, output_type) -> data`

Runs one pipeline step in an isolated `exec` scope that has `mne`, `numpy`, and the pipeline functions preloaded.

**Input injection:**
- `DataType.ICA`: unpacks `ICASolution` into `scope["ica"]` and `scope["raw"]`.
- Other types: injects data under its scope variable name (`raw`, `epochs`, `evoked`).

**Output extraction:**
- `DataType.ICA`: re-bundles `scope["ica"]` and `scope["raw"]` into a new `ICASolution`.
- Other types: reads the scope variable for `output_type`.

Custom actions (empty `func_defs`, arbitrary `call_site`) degrade to a direct `exec`.

## Project (`project.py`)

`project.json` is the source of truth for a project. Key classes:

### `Project`

Top-level container. Fields: `name`, `participants: list[Participant]`, `has_default_pipeline: bool`, `created_at: str`.

### `ProjectContext`

Carries context from the project page into a `PreprocessingPage` instance:

| Field              | Type          | Purpose                                                       |
|--------------------|---------------|---------------------------------------------------------------|
| `project`          | `Project`     | Parent project                                                |
| `project_dir`      | `Path`        | Project directory on disk                                     |
| `participant`      | `Participant` | Participant being preprocessed                                |
| `session`          | `Session`     | Session being preprocessed                                    |
| `data_files`       | `list[Path]`  | Resolved file paths to load (single run or merged list)       |
| `run_index`        | `int \| None` | Run index within the session, or `None` for merged runs       |
| `on_status_update` | `Callable`    | Called when the preprocessing page closes with the new status |

### `Participant`

Fields: `id: str`, `sessions: list[Session]`, `notes: str`.

`participant_status`: property that aggregates all session statuses.

### `Session`

| Field                 | Type                | Purpose                                               |
|-----------------------|---------------------|-------------------------------------------------------|
| `id`                  | `str`               | Short session identifier                              |
| `data_files`          | `list[str]`         | Raw EEG file paths, one per run                       |
| `merge_runs`          | `bool`              | Concatenate all runs before processing                |
| `processed_files`     | `list[str]`         | Exported `.fif` paths, parallel to `data_files`       |
| `has_custom_pipeline` | `bool`              | Session uses its own pipeline override                |
| `pipeline_hashes`     | `list[str]`         | Hash of the canonical pipeline used per run           |
| `status`              | `ParticipantStatus` | Stored status; only trusted for `ERROR` and `RUNNING` |
| `error_msg`           | `str`               | Last pipeline error message                           |

### `ParticipantStatus`

Enum used for display in the project page:

| Value        | Meaning                                        |
|--------------|------------------------------------------------|
| `PENDING`    | No runs processed yet                          |
| `RUNNING`    | Processing currently in progress               |
| `DONE`       | All runs processed with the current pipeline   |
| `STALE`      | Processed, but pipeline changed since last run |
| `INCOMPLETE` | Some runs done, some pending or stale          |
| `ERROR`      | Processing failed                              |

## Data Store (`data_store.py`)

Large intermediate `Raw`, `Epochs`, and `Evoked` objects are checkpointed to disk between pipeline steps using FIF files in a temporary directory. An in-memory LRU cache avoids redundant disk I/O during sequential execution and step navigation.

The store is invalidated automatically when the action list changes so stale checkpoints are never used.
